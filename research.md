## Anchor question — answered (2026-07-13)

**What feedback were you missing that killed past abandoned tracks?**

Learner has ADHD. Four named failure modes, none of them memory-decay:

1. **Novelty-dopamine propulsion.** Start is fueled by the rush of imagining where the project could go (e.g. "Rust WebGPU app that turns into a game engine people might use"). That rush is structurally transient; when it fades there's nothing to replace it.
2. **Static, brain-mismatched resources.** Learning materials are too static, in the wrong order/format for the learner's brain → boredom.
3. **Executive dysfunction on re-entry.** Picking the project back up, doesn't know where to start.
4. **Yak-shaving traps.** Tangential setup/tooling that isn't core learning kills motivation.

**Reframe:** the completion bottleneck is *upstream of memory*. Retention-oriented traditions (spaced repetition, retrieval practice, mastery gating) solve a problem the learner doesn't yet have. They may become relevant *after* the completion problem is solved — flagged as future concern, not v1.

## Architecture implications (filled in after verdicts)

**Synthesized from Round 1 + Round 2 verdicts (2026-07-13).** These are the load-bearing mechanisms the package must provide. Not a final architecture — the design layer comes after Round 3+ verdicts, but these are now fixed constraints:

1. **Per-track "current edge" marker** (from #4 adopt). The package always knows the specific thing just beyond current ability. Every session orients around it. Distinct from mastery gating (#3) — edge is about the *next* hard-but-doable thing, not a barrier to advance.

2. **Always-on pre-computed next concrete action** (from #7 adopt). The package holds a waiting next-action so re-entry is "execute" not "figure out." This is what `socrates-plan` was reaching for but didn't survive to re-entry. Persistence across sessions is the hard part.

3. **Flow-shaped sessions** (from #8 adopt): clear sub-goal + immediate feedback + calibrated difficulty. The session tool produces flow conditions, not just counts minutes. The verify step is the immediate-feedback half. Calibrated difficulty connects back to #1 (edge marker).

4. **Retrieval-shaped verify** (from #2 adapt): verify should require *produce-before-check*, not recognition. Reading alone doesn't encode — the learner must manipulate and apply. This is a property of the verify mechanism, not a standalone tool.

5. **Cue → routine → reward loop** (from #10 adopt): recurring cue fires, triggers low-friction routine (open track → read next action → start timer), logs a small reward. `socrates-timer` is the routine half; cue and reward are missing. The cue is the novel piece — without it, re-entry still depends on motivation.

6. **Scope guard with TWO modes** (from #11 adapt — the reshape):
   - **Mode A — tangential drift:** honest mirror ("you've spent 40 min on X; the current edge is Y"), no block, defer to a deferred-yaks list so it doesn't feel like a loss.
   - **Mode B — required-but-painful setup:** when the yak is legitimately required (WebGPU env, ML fundamentals toolchain, etc.) and not just avoidable drift, the package *generates the scaffolding itself*. This is the strongest new signal from Round 2 — the package is not just a mirror, it's an active scaffold-eliminator for setup pain that has no good tutorial. Implies a project-scaffolding capability in the package, possibly the largest single feature.

7. **Permissive-but-honest gating** (from #3 adapt, weakly): gates surface core vs tangential, don't block. Reinforces #6 Mode A.

8. **Lightweight end-of-session reflection** (from #5 adopt): NOT a journal — a structured micro-step at session end that updates (a) the edge marker #4 — did we cross it? is a new edge visible? — and (b) the next-action #2 — what's the concrete first move next time. This is the feedback path that keeps #1 and #2 alive across sessions; without it, the pre-computed next-action goes stale and re-entry breaks down again (the exact failure of `socrates-plan`). Double-loop (Argyris) is a *sparingly-triggered* variant: only when a track has stalled across N sessions, surface "is the *goal* wrong, not just the approach?" — the signal you're chasing a goal that isn't yours.

9. **Outcome-as-compass, process-as-contract, edge-as-measure** (from #6 adopt): each track holds an outcome-statement (the vision that fired the start) + a process-target (the cue/routine contract from #5). Progress is measured against the edge #1, NOT the outcome. The outcome is kept *visible* (so the vision doesn't decay into guilt) but *never gated on* (gating on a distant outcome is demoralizing). **Critical reshape from Round 3 `felt_absence_3`:** the outcome-statement is a *living, revisable* compass — the reflection loop #8 can revise it as the vision sharpens. It is NOT a fixed SMART target. The learner's vision is "fluid and dynamic," so the package must treat it as such.

10. **Cross-track interleaving** (from #9 adapt): vary the surface across tracks (not within one) to keep the novelty/flow source from collapsing. Addresses failure mode #1 from a third angle alongside flow #3 and habit #5. Within-track interleaving stays deferred to post-completion (same bucket as spaced repetition). Mechanism is light: the cue/routine #5 can pick *which* track today, not just *whether* to sit down — keeps "which thing today?" fresh without depending on the novelty rush.

11. **Material ingestion + edge-driven sequencing** (from ✦ adopt — the cross-cutting thread): a "material ingestion" step — feed in a book/course/docs, the package decomposes it into learnable units, tags each with prerequisite/difficulty, and the plan tool sequences them against the current edge #1 rather than the author's order. The socratic skill produces the questions; the plan tool produces the sequence. This is the mechanism that makes #1 (edge) and #2 (next-action) work against *external* material instead of just self-defined tracks. **Critical reshape from Round 3 `felt_absence_3`:** ingestion is *iterative and revisable*, NOT a one-shot decomposition. The learner "doesn't always know exactly what they want to learn" but holds a "fluid and dynamic" general vision — so the decomposition must be re-runnable, with the reflection loop #8 revising the sequence as the vision sharpens. This is the strongest new architecture signal from Round 3, paralleling the scaffolding-generator reshape to #11/scope-guard from Round 2.

**Open architecture questions (now narrowed — research phase complete):**
- *Data model:* what does a "track" record look like such that it holds the edge #1, next-action #2, outcome-compass #9, process-contract #5, deferred-yaks #6, and ingested-material-unit-graph #11 — and persists across sessions?
- *Persistence:* how does the package survive the re-entry moment? `socrates-plan` didn't. This is the hardest engineering problem and the highest-leverage one (it's what #2, #5, #8, #10 all depend on).
- *Tool boundaries:* which of these mechanisms are pi tools (commands the learner invokes), which are skills (socratic questioning, reflection prompts), and which are background loops (the cue, the stall-detector for double-loop)? The scaffolding-generator #6 Mode B and the material-ingestion #11 are the two largest feature candidates and likely both tool+skill hybrids.
- *Iteration loop:* how do #8 (reflection) and #11 (ingestion-revision) compose? The reflection loop updates the edge; the ingestion-revision updates the sequence against the edge. Are they one step or two?

**Next phase:** turn these 11 mechanisms + the 4 open questions into a concrete pi-package design.

## Literature verification (2026-07-13)

After the design phase converged, the verdict trail was checked against current literature via live web search (9 searches covering the anchor failure modes + all 11 traditions + the cross-cutting thread). **Result: every verdict holds.** Three traditions received research-grounded *refinements* (not reversals) — recorded below. The single strongest find is a 2026 book chapter that reads as a near-direct literature precedent for the package's thesis.

### Anchor + failure modes — confirmed

- **ADHD novelty-dopamine + hyperfocus paradox.** The "Purpose-Driven Motivation Hypothesis" (Ortega, 2026; Zenodo preprint) directly names the "hyperfocus paradox — extraordinary sustained attention on self-selected activities in individuals who struggle with externally-imposed tasks" and proposes dopamine is "differentially sensitive to (1) alignment between activity and intrinsic interests, and (2) perceived progress toward meaningful goals." This is exactly failure mode #1 (novelty-dopamine propulsion) + the mechanism the package targets (alignment via the outcome compass #9 + perceived progress via the edge #4). Herd, Mingus & O'Reilly (CCN Lab) link dopamine directly to self-directed learning. Failure modes #1 + #2 are grounded, not story.
- **Executive dysfunction on re-entry — the strongest precedent.** McLeod, Miller-Felton & Kim, *Exploring AI Support for Executive Function Deficiencies and Self-Regulated Learning in Adult Learners with ADHD* (2026, Dominion University College book chapter, DOI 10.59668/2579.27201) is almost a direct description of learn-pi's thesis. Key quote: *"the performance phase of Zimmerman's SRL framework poses the greatest challenges due to time management and task initiation. However, issues transitioning between the forethought phase, as participants set academic goals but lacked confidence in following through. In addition, there is a disruption between the reflection and forethought phases as participants assess academic performance but do not adjust learning strategies."* This is precisely the three breaks the package's design fixes: the `next_action` field bridges forethought→performance; `session_start` widget renders it before executive function is needed; `/learn-reflect` closes reflection→forethought. The chapter further names *"AI-driven tools can provide features such as prompting, notifications, feedback, recognizing and responding to emotions, and visualizing performance"* — a near-1:1 mapping to the cue (#10), the dashboard widget, the verify step (#3), and the reflection log (#8). The 2026 arXiv paper *Scaffolding Metacognition with GenAI* (2602.09381) adds the caution: GenAI can inhibit metacognition if it outsources the problem — validates the socratic-method skill's "ask before telling" contract rather than a homework-machine mode.

### Traditions — confirmed with three refinements

- **#4 Deliberate practice — confirmed.** Mirlohi & Herman (2024, *The Role of Challenge-Skills Balance in Task Engagement*, DOI 10.4324/9781003406419-8): *"If the challenge level of a task is too high, the performer is more likely to give up in frustration. Accordingly, if the difficulty level is too low, the performer is more likely to become bored and disinterested."* Maps exactly to failure mode #2. The structural-learning.com synthesis (citing Ericsson 2008, Hattie & Timperley 2007) confirms "tasks set just beyond current performance" + "immediate, specific, actionable feedback" as the critical accelerators. No refinement needed.
- **#7 SRL (Zimmerman) — confirmed.** Three independent 2024-2025 sources (IntechOpen *Beyond Symptoms: ADHD and the Role of SRL*; ERIC EJ1479884 flipped-classroom study; Zimmerman 2002 *Becoming a Self-Regulated Learner*) converge on the forethought/performance/reflection cycle. The DiBenedetto & Zimmerman (2010) microanalytic study is cited as the empirical backbone. No refinement.
- **#8 Flow — REFINED.** *The Feedback Loop of Flow* (AISel, DOI 10.17705/1thci.00172) establishes causally that **feedback must be task-relevant** (tied to the proximal goal) to induce flow — not merely immediate. *"Task-relevant feedback communicates how well users are performing actions that make progress towards the goal of the optimally challenging task… the feedback loop of flow is a recursive process, which suggests task-relevant feedback must be presented continuously and for each subtask."* **Refinement to verdict #8 / mechanism #3:** the verify step is not just "immediate feedback" — it must be feedback tied to the *current edge's proximal sub-goal*. A generic `cargo test` pass/fail is weaker than a verify that names what the edge was. Implication for the design: when `/learn-reflect` records `edge_crossed`, it should also capture *which sub-goal the verify spoke to*, so the feedback loop is provably task-relevant, not just timed. The 2024 *Journal of Happiness Studies* response-surface analysis (Kotler et al.) confirms challenge-skill balance remains the dominant antecedent (β=.245, p<.001).
- **#10 Implementation intentions / habit loops — confirmed and strengthened.** Sheeran, Listrom & Gollwitzer (2024, *The when and how of planning*, DOI 10.1080/10463283.2024.2334563) — a meta-analysis of **642 independent tests** — finds if-then plans effective for cognitive, affective, and behavioural outcomes (d = .27–.66). Critically: *"Effect sizes were larger when plans had a contingent (if-then) format, participants were highly motivated to pursue the goal, and plans were rehearsed."* The **"rehearsed" moderator is a design plus for the hard cue**: the launchd notifier script reads the current `next_action` fresh at fire time and surfaces it in the notification body — the cue *is* a rehearsal of the if-then plan, not just a ping. This validates Fork E (hard cue) beyond the original rationale: the notification carrying the next action is itself the rehearsal loop the meta-analysis identifies as a moderator.
- **#5 Reflection (Kolb/Argyris) — confirmed.** *Multi-stage Team Projects* (OBTR 2025, DOI 10.1177/10525629241313196) applies double-loop peer feedback in 2024-2025 pedagogy. The MSP Reflection Methods handbook distinguishes single/double/triple-loop cleanly. No refinement — the sparing use of double-loop (only on stall) matches the literature's caution that double-loop is for "when expected results cannot be achieved by adapting the rules."
- **#6 Goal-setting (outcome vs process) — confirmed and nuanced.** Pham & Freund (2025, *European Journal of Psychology of Education*, DOI 10.1007/s10212-025-00995-x): *"Outcome focus generally inversely correlates with engagement and diminishing intrinsic motivation."* Directly supports verdict #6's "outcome as compass, never gated." **Nuance worth recording:** the same study found goal-focus *congruence* doesn't simply maximize engagement — the *direction* of incongruence matters (process-focus with a visible outcome compass may be the sweet spot, not "have both equally"). The package's design (outcome visible + process-as-measure + edge-as-true-measure) lands in this nuance correctly, but it's worth noting the research doesn't give a clean "both is best" endorsement.
- **#9 Interleaving — REFINED.** Brunmair & Richter (2019, *Psychological Bulletin*, meta-analysis of 59 studies / 238 effect sizes): overall interleaving benefit g = .42, BUT heavily material-dependent — paintings g = .67, math g = .34, **expository texts g = .21 (nonsignificant), words g = −.39 (blocking wins).** Li, Liu, Xu & Yi (2024, *MIS Quarterly*): *"related interleaving"* (mixing structurally related topics) outperforms both no-interleaving and unrelated interleaving, **strongest for weaker learners.** **Refinement to verdict #9 / mechanism #10:** cross-track interleaving should be **structurally related cross-track**, not random variety. "Vary the surface" was too vague — the research says vary it with tracks that share underlying structure, and the benefit is largest for the learner who struggles most. Implication for the design: when the cue/session-start offers "which track today?", it should prefer *structurally adjacent* tracks (e.g. rust-webgpu ↔ rust-rag-learn share the Rust substrate), not arbitrary rotation. This becomes a v1.1 feature for `/learn-ingest`-derived tracks (the material graph carries prerequisite/structure tags that enable related-interleaving selection).
- **#2 Retrieval practice (verify shape) — confirmed.** Sana & Yan (2022) combined interleaving + retrieval practice in high-school science with benefits for both memory and transfer. The "produce-before-check" property of the verify mechanism matches the retrieval-practice tradition's active-recall requirement.
- **#3 Mastery gating — confirmed.** The permissive-but-honest adaptation aligns with the literature's caution against rigid gating for learners already struggling with initiation.

### Cross-cutting personalized curriculum — REFINED (the strongest validation)

Three 2026 papers directly validate and sharpen the Round 3 reshape (iterative, revisable ingestion):

- **PathBuilder** (ACL 2026, aclanthology.org/2026.acl-demo.50.pdf) — a quality-controlled LLM system for personalized learning pathways, deployed with 179 active users. Key architectural finding: *"LLM-generated instructional content can suffer from hallucinations or conceptual drift. To mitigate these risks, PathBuilder employs a Retrieval-Augmented Generation (RAG) architecture grounded in a curated knowledge base."* **Refinement to mechanism #11:** `/learn-ingest` (v1.1) must be **RAG-grounded** — the decomposition must retrieve from the actual ingested material, not free-generate from the model's training corpus. The material's text is the knowledge base; the LLM sequences and tags against it. This is the research-grounded version of the Round 3 "iterative, revisable" reshape.
- **Hierarchical Knowledge Graph-Augmented LLM for Learning Path Generation** (2026, DOI 10.1145/3806262.3806285) — uses a hierarchical KG + *"self-feedback-based iterative method for high-quality LPG"* with measurable gains in knowledge coverage, personalization, and pedagogical coherence. **Directly validates the iterative-revision loop:** the reflection loop #8 re-running ingestion sequencing against the edge is the same pattern as the paper's self-feedback iteration.
